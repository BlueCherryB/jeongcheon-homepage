# CMS Decisions

## 1. Purpose

This document records the finalized initial decisions for integrating Sanity with the Jeongcheon Law Office website. It prepares the project for Task #021, which will initialize Sanity Studio, without installing Sanity or changing runtime application behavior in Task #020.

## 2. Final Decisions

- Sanity Studio will live in the existing repository as an independent application.
- The Studio directory will be `studio/`.
- The root project remains the Next.js website.
- Studio will not be embedded in the Next.js App Router.
- No `/studio` route will be created under `src/app`.
- No workspace or monorepo tooling will be introduced initially.
- The planned Sanity display name is `Jeongcheon Law Office`.
- The Sanity project ID must be generated by Sanity and must not be invented in documentation.
- The initial dataset will be `production`.
- Published website content will use a public dataset.
- The public website should read published content without a token initially.
- Sanity must contain only anonymized content approved for public website publication.

## 3. Repository Structure

Planned structure:

```text
jeongcheon-homepage/
  src/
  public/
  docs/
  studio/
  package.json
```

Root application:

- Next.js website.
- Reads published content.
- Contains CMS query and mapping layers.
- Does not contain Studio UI.

`studio/`:

- Independent Sanity Studio application.
- Contains schemas and desk structure.
- Has its own `package.json`.
- Has its own build and development commands.
- Can be deployed separately.

Do not create `studio/` until Task #021.

## 4. Project and Dataset Naming

Sanity display name:

```text
Jeongcheon Law Office
```

Sanity project ID:


The project ID must be copied from Sanity after project creation. It must not be invented or hard-coded before then.

Initial dataset:

A separate `development` dataset is deferred. It may be introduced later if schema experimentation or isolated content testing becomes necessary.

## 5. Dataset Visibility and Token Policy

Initial dataset visibility:

- Public dataset for published website content.

Public website policy:

- Read published content without a Sanity API token initially.
- Never expose a write token.
- Never expose a preview token.
- Never query drafts during normal production requests.

Future preview policy:

- May use a server-only read token with draft access.
- Must not expose preview tokens to the browser.
- Must be implemented in a separate task.

Planned website environment-variable contract:

```text
NEXT_PUBLIC_SANITY_PROJECT_ID=20zyfjea
NEXT_PUBLIC_SANITY_DATASET=production
NEXT_PUBLIC_SANITY_API_VERSION=2025-02-19
SANITY_API_READ_TOKEN=<SERVER_ONLY_READ_TOKEN_FOR_FUTURE_PREVIEW>
```

Rules:

- `NEXT_PUBLIC_SANITY_PROJECT_ID` may be exposed publicly.
- `NEXT_PUBLIC_SANITY_DATASET` may be exposed publicly.
- `NEXT_PUBLIC_SANITY_API_VERSION` may be exposed publicly.
- `SANITY_API_READ_TOKEN` must remain server-only.
- The initial published-content implementation should not require `SANITY_API_READ_TOKEN`.
- The token is reserved for future preview or authenticated draft access.
- Do not define a write token for the website.
- `NEXT_PUBLIC_SANITY_API_VERSION` is fixed to `2025-02-19`.
- Do not use `latest`, `today`, or any runtime-generated date for the Sanity API version.
- Use the same variable names in local `.env.local`, Vercel settings, and Cloudflare settings.
- Use `@sanity/client` directly for the initial read-only query layer.
- Do not introduce `next-sanity` for the initial published-content client.
- Published reads use the published perspective without a token.
- Case Study reads bypass the Sanity CDN so webhook path revalidation fetches fresh published content immediately after publish, update, or delete events.
- User-derived values such as slugs must be passed to GROQ through parameters.
- Raw CMS response types stay separate from future application-facing content types.
- Application-facing content types live outside the CMS layer.
- UI code should consume application content types and should not depend on raw Sanity response types.
- Case Study mappers translate CMS field names into website field names.
- The Sanity Case Study field `legalIssues` maps to the application field `issues`.
- A single invalid Case Study detail record should fail loudly at the mapper boundary.
- Case Study collection mappers may skip invalid records so listings can still render valid published content.
- During migration, the application content API reads Sanity first and falls back to existing local Case Study data when Sanity is empty or unavailable.
- If Sanity and local data contain the same slug, Sanity content takes priority.
- Static slug discovery may combine Sanity and local slugs, with duplicates removed, until the CMS migration is complete.
- The preview/draft client remains deferred and must be server-only when introduced.
- Case Study images may use the Sanity CDN for published content.
- Website UI must use application-facing image types, not raw Sanity image response types.
- Sanity image URLs should be derived in a focused helper and should fall back safely when required public Sanity environment variables are unavailable.
- Sanity crop metadata may be applied as a CDN source rectangle, but final fixed-ratio card and hero cropping should remain in CSS so Sanity hotspot values can control `object-position`.
- The website may allow `https://cdn.sanity.io/images/**` in Next.js image remote patterns.
- Do not add `@sanity/image-url` unless the manual helper becomes insufficient for approved crop/hotspot behavior.
- Sanity publish/update/delete webhook revalidation uses the server-only `SANITY_REVALIDATE_SECRET`.
- The webhook sends the secret in the `x-sanity-revalidate-secret` request header to `/api/revalidate/sanity`.
- The revalidation secret must be configured in Vercel Preview and Production and must never be exposed with `NEXT_PUBLIC_`.

Do not add secrets to `.env`, `.env.local`, Git-tracked files, or documentation examples with real values.

The committed `.env.example` may include the public project ID because it is already committed in the Studio configuration and is not a secret. It must never include a real API token.

The Studio currently keeps its project ID and dataset in `studio/sanity.config.ts` and `studio/sanity.cli.ts`. Do not move Studio configuration into website environment variables unless a later deployment task requires it.

## 6. Studio Deployment Boundaries

The website and Studio may share the same Git repository, but they must remain independently buildable and deployable.

Hosted Studio deployment:

- Studio hostname: `jeongcheon-law-office`.
- Studio URL: `https://jeongcheon-law-office.sanity.studio/`.
- Deployment command: run `npm run deploy` from `studio/`.
- Studio uses Sanity authentication and connects to project `20zyfjea` with dataset `production`.
- Do not expose Studio credentials or API tokens in the website.
- A public website administrator link is deferred to a later task.

Do not:

- Embed Studio in `src/app`.
- Create a Next.js `/studio` route.
- Couple the website build to the Studio build.
- Introduce Turborepo, Nx, pnpm workspaces, or npm workspaces for the initial setup.

## 7. Portable Text Scope

Use Portable Text from the beginning for these Case Study fields:

- Case overview.
- Main legal issues.
- Jeongcheon's response.
- Case result.

Conceptual reusable schema object:

```text
blockContent
```

Allowed styles:

- `normal`
- `h2`
- `h3`

Allowed lists:

- `bullet`
- `number`

Allowed marks:

- `strong`
- `link`

Allowed link values:

- `https://`
- `http://`
- `mailto:`
- relative internal paths beginning with `/`

Do not add advanced blocks initially:

- Inline images.
- Tables.
- Video embeds.
- Custom callout blocks.
- Arbitrary HTML.
- Code blocks.

## 8. Content Privacy Rules

Sanity must contain only content that is approved for public website publication.

Do not store:

- Client names unless explicitly approved for publication.
- Personally identifiable client information.
- Resident registration numbers.
- Client phone numbers.
- Private evidence.
- Internal case files.
- Unredacted court documents.
- Confidential legal strategy.
- Attorney-client privileged material.

Case-study content must be anonymized and reviewed before publication.
